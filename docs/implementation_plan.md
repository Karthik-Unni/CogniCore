# CogniCore Character Intelligence Framework & Silent Hollow 3D Demo

CogniCore is an open-source, reusable Character Intelligence Framework designed to enable persistent AI characters with memory, emotions, relationships, goals, planning capabilities, social reasoning, and autonomous behavior. 

This plan details the implementation of the CogniCore SDK, a real-time multi-agent simulation server (FastAPI), and a stunning developer dashboard + 3D simulation environment (Silent Hollow) using React, Three.js, and CSS modules.

## What Makes CogniCore Interesting

The core philosophy of CogniCore is: **Predictable rules, unpredictable stories.**

The framework focuses on deterministic mechanics combined with agent autonomy to produce genuine emergent narratives.
- **What is predictable**: The rules of emotions, relationship changes, memory storage, rumor degradation, and planning. You know exactly *how* a character processes a lie, how a rumor mutates when told by a liar, or how fear decays.
- **What is unpredictable**: The story itself. You do *not* know who will form alliances, who will become suspicious of whom, who will weaponize which secrets, or who will eventually be accused of the crime until the simulation runs.

If every run of *Silent Hollow* ended with the same murderer, same suspects, and same conversations, it would be a scripted game. When every run produces a different, believable social outcome while strictly adhering to the framework's mathematical rules, we have built a genuine multi-agent character intelligence engine.

## Evaluation & Consistency Metrics

To verify that the emergence is believable rather than chaotic, the framework implements several automated validation metrics:

### 1. System Memory Consistency
*   **Definition**: NPCs must not contradict their own recorded memories or alibis during conversation, and long-term consolidation must preserve core facts.
*   **Metric**: Accuracy of NPC statements against the SQLite vector database. Alibis generated by the planning engine must align with the agent's spatial timeline log (e.g. if an agent was at the Blacksmith at tick 10, they cannot claim they were at the Tavern at tick 10 without their planning engine explicitly flagging it as a "conscious lie").

### 2. Personality Consistency
*   **Definition**: Actions, dialogue, and plans must match the agent's initial personality vector (e.g., `Greed`, `Honesty`, `Aggression`).
*   **Metric**: Alignment score between chosen actions and personality weights. For instance, a character with high `Honesty` should have a very low probability of lying unless the planning engine justifies it with a critical survival goal.

### 3. Goal Adherence
*   **Definition**: NPC actions must actively contribute to the completion of their current top-priority goal.
*   **Metric**: Path-to-Goal ratio. We measure the percentage of tick actions that directly or indirectly attempt to resolve the active goal (e.g., if the goal is `Hide Crime`, checking if actions like `Bribe Guard`, `Move Evidence`, or `Lie to Player` are executed over unrelated activities like `Gather Herbs`).

### 4. Social Consistency
*   **Definition**: Relationship weights (trust, fear, friendship) must change logically based on social transactions and match the character's behavior.
*   **Metric**: Interactive alignment. If `Trust` with Character B is -0.8, Character A should refuse to share secrets with B, act defensively in conversations, and prioritize accusing B if suspicious, unless engaging in strategic manipulation.

## Proposed SDK Architecture (`cognicore`)

The SDK is written in Python, designed as a modular library that can be imported into any application or game engine (via a WebSocket/HTTP adapter).

```mermaid
graph TD
    A[Agent Runtime Loop] --> B[Observation System]
    A --> C[Memory System (Short/Long/Social/Secret)]
    A --> D[Emotion Engine]
    A --> E[Relationship Graph]
    A --> F[Goal & Planner Engine]
    A --> G[Rumor & Secret Management]
    
    C --> H[RAG & SQLite Vector Store]
    F --> I[LLM Layer & Local Rule Fallback]
```

### Component Details

#### 1. Core Agent Loop (`cognicore/agents/`)
Every agent runs on a heartbeat or event-driven step:
1. **Observe**: Receive external events (sensory inputs, player statements, physical location changes).
2. **Retrieve Memory**: Query the RAG layer for relevant memories based on the observation context.
3. **Update Emotions**: Calculate numerical emotional updates based on memory importance and event type.
4. **Update Relationships**: Adjust relationship values (trust, fear, friendship) relative to the actors involved.
5. **Evaluate Goals**: Recalculate goal weights and select the highest priority goal.
6. **Generate Plan**: Create a series of actions (e.g. walk to tavern, seek information, lie to player, share rumor) to satisfy the goal.
7. **Choose Action**: Pick the immediate next action.
8. **Execute Action**: Dispatch the action to the world simulation.
9. **Reflect**: Periodic review of short-term memories to consolidate them into long-term summary memories.

#### 2. Memory System & RAG (`cognicore/memory/`, `cognicore/rag/`)
- **Memory Types**:
  - *Short-Term*: Active sliding window of recent interactions.
  - *Long-Term*: Significant events (high emotional impact or relevance).
  - *Social Memory*: Specific mappings of what this NPC knows/believes about other characters.
  - *Secret Memory*: Hidden information that has a "reveal risk" score.
- **RAG Vector Storage**:
  - A lightweight, built-in SQLite-based vector store implementing a pure-Python cosine-similarity or TF-IDF algorithm. This guarantees a zero-dependency setup.
  - Support for external vector databases (ChromaDB or Qdrant) via a configuration flag.
  - Each memory node includes: `timestamp`, `content`, `importance` (1-10), `emotional_impact` (valence/arousal), `confidence`, and `tags` (related entities).

#### 3. Emotion Engine (`cognicore/emotions/`)
- Numerical modeling of 7 key emotional dimensions: `anger`, `fear`, `suspicion`, `happiness`, `trust`, `guilt`, `confidence`.
- Updates occur using an emotion transition matrix (e.g., being accused increases `anger` and `fear`; sharing a secret increases `trust`).
- Natural decay function running on world ticks to return emotions toward a baseline over time.

#### 4. Relationship Graph (`cognicore/relationships/`)
- A directed social graph mapping relations between all agents.
- Links store: `trust`, `respect`, `fear`, `friendship`, `rivalry`, and `loyalty` (values normalized between -1.0 and 1.0).
- Social interactions dynamically update these values (e.g. lying decreases `trust`, helping increases `friendship`).

#### 5. Goal & Planning Engine (`cognicore/goals/`, `cognicore/planning/`)
- **Goals**: Represented as hierarchical objects with static weights and dynamic multipliers based on world state (e.g., *Self-Preservation* priority spikes when suspicion rises).
- **Planner**:
  - *LLM Planner*: Prompts the LLM with the agent's current state, recent memories, relationships, and active goals to generate a reasoning trace and a sequence of simulation actions.
  - *Rule-Based Planner*: A utility-based system matching active goal tags to a library of available actions, making the framework run efficiently and predictably without external APIs.

#### 6. Secrets & Rumors (`cognicore/secrets/`, `cognicore/rumors/`)
- **Secrets**: Encapsulate facts or events with a designated *Secrecy Level* and *Expose Penalty*. Characters try to minimize exposure but may trade secrets if cornered, blackmailed, or when trust exceeds a threshold.
- **Rumors**: Pieces of information that spread from NPC to NPC during interactions. 
  - Rumors have a *Fidelity* score.
  - Mutate over time (NPCs with low honesty or high suspicion may exaggerate or misremember details when retelling them).
  - NPCs evaluate rumor credibility based on their relationship/trust with the source.

#### 7. LLM Provider Layer (`cognicore/llm/`)
- A unified interface allowing plug-and-play LLM providers.
- Supports: `OpenAI`, `Gemini`, `Anthropic`, `Ollama`, and `Mock` (for local deterministic operation).

## Showcase Application & Dashboard (`silent_hollow_demo`)

A murder mystery simulation in a stylized village. 

```
silent_hollow_demo/
├── backend/
│   ├── server.py              # FastAPI server running agent loops
│   ├── world.py               # Village simulation grid and locations
│   └── scenario.py            # Murder setup, dynamic killer assignment
├── frontend/
│   ├── src/
│   │   ├── components/        # UI components (Agent inspector, case board)
│   │   ├── three/             # Three.js 3D rendering (Village, pathfinding, animations)
│   │   ├── styles/            # Vanilla CSS stylesheets (Glassmorphism, dark theme)
│   │   └── App.jsx            # Main React layout
│   ├── index.html
│   └── package.json
```

### The "Silent Hollow" Simulation
- **Dynamic Setup**: A random villager (e.g., Arthur the Scribe) is found murdered. One of the 8 characters is dynamically chosen as the killer, given a motive, a secret (the crime), and a hidden weapon/evidence placement.
- **Agent Roles**:
  1. *Mayor Alden* (Town Hall)
  2. *Guard Katherine* (Guardhouse)
  3. *Merchant Marcus* (Shop)
  4. *Blacksmith Dennis* (Forge)
  5. *Doctor Clara* (Clinic)
  6. *Innkeeper Elena* (Tavern)
  7. *Farmer Silas* (Farms)
  8. *Hunter Gerald* (Forest Cabin)
- **Agent Behavior**: Characters move between locations, interact in groups, gossip (spreading rumors about the murder), and update their alibis and suspicions based on what they learn.

### Developer Dashboard & Player HUD
- **3D Isometric Viewport**: Rendered using Three.js. Shows the 3D village with buildings, paths, trees, and NPCs moving around. Hovering or clicking an NPC highlights them and shows their paths.
- **Observability Inspector**:
  - **Emotions Panel**: Visual radar chart or horizontal meters of current emotions.
  - **Social Graph**: Interactive node-link diagram showing lines of trust/friendship/rivalry between villagers.
  - **Memory Log**: Filterable timeline of the agent's short-term, long-term, and secret memories.
  - **Goal & Plan Display**: Shows current target goal, active sub-tasks, and LLM reasoning trace.
  - **Rumors & Secrets**: Lists all rumors known (and their mutations) and secrets held.
- **Player Detective Mode**:
  - Click on locations to "inspect" them for physical clues (letters, drops of blood, objects).
  - Interact/Interrogate NPCs via chat interface. You can ask for their alibi, ask what they know about other villagers, show evidence to provoke reactions (increases fear or anger), or accuse them.
  - **Accuse Button**: Accuse a character. If you have enough evidence, they confess. If you accuse an innocent, it damages your reputation or leads to a different ending.

## Verification Plan

### Automated Tests
- Unit tests for the emotion updates, relationship graph modifications, and SQLite RAG similarity.
- Integration tests checking rumor propagation (verify a rumor starts at NPC A and spreads to NPC D after 3 interaction ticks).
- Simulation test suite running a fully automated 50-tick game run to ensure no deadlocks or exceptions in the agent loop.

### Manual Verification
- Visual validation of the 3D viewport pathfinding and characters moving to their locations.
- Interrogating the characters through the player interface to confirm alibis align with their actual logs and memories.
- Verifying the real-time updates of the developer dashboard panels upon clicking different NPCs.
